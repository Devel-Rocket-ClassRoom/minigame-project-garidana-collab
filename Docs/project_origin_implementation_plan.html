Project Origin 앞으로 구현할 큰 시스템 정리

노션의 Project Origin 3주 프로젝트 문서와 태스크 트래커 내용을 바탕으로, 남은 구현 범위를 큰 시스템 단위로 재분류하고 Unity에서 어떤 구조로 구현하면 좋은지 정리한 문서입니다.

프로젝트 ka_origin / 김경일의 모험: Origin
기간 2026.05.19 - 2026.06.08
장르 3D 오픈월드 액션 판타지 RPG
핵심 방향 3주 안에 완성 가능한 선형 진행형 액션 RPG

전체 구현 전략

모든 기능을 독립적으로 만들기보다, 데이터 기반 시스템을 먼저 잡고 그 위에 콘텐츠를 얹는 방식이 좋습니다. 플레이어, 몬스터, 아이템, 퀘스트, 웨이포인트는 서로 연결되므로 ScriptableObject와 Manager 계층을 기준으로 의존성을 정리해야 합니다.

추천 우선순위: 플레이어 조작과 전투 루프를 먼저 안정화하고, 그다음 몬스터/보상/퀘스트를 연결한 뒤, 인벤토리와 장비 비주얼 교체는 최소 기능부터 완성하는 것이 일정 리스크를 줄입니다.

큰 시스템별 구현 방향

Core플레이어 입력 / 이동 / 카메라

Input System을 중심에 두고 입력 해석과 실제 이동 처리를 분리합니다.

  • PlayerInputReader: Move, Attack, Dash, Sprint, Interact 같은 입력 이벤트만 담당합니다.
  • PlayerMovement: CharacterController 또는 Rigidbody 기반 이동, 대시 쿨다운, 중력 처리를 담당합니다.
  • PlayerCamera: 쿼터뷰 고정 카메라를 기본값으로 두고, 줌은 옵션으로 유지 여부를 결정합니다.
  • Animator 파라미터는 이동 속도, 대시 상태, 공격 상태 정도로 단순하게 유지합니다.

Combat전투 / 공격 강화

스킬 시스템 대신 기본 공격 강화 4단계를 쓰는 방향이므로, 공격 데이터 테이블을 먼저 설계하는 것이 중요합니다.

  • AttackData ScriptableObject: 데미지, 범위, 쿨타임, 이펙트 색상, 히트박스 크기, 애니메이션 트리거를 보관합니다.
  • PlayerCombat: Space 입력, 3단 콤보, 콤보 리셋 시간, 공격 중 이동 제한을 처리합니다.
  • Hitbox: 공격 애니메이션 이벤트 시점에만 Collider를 켜서 몬스터에게 데미지를 전달합니다.
  • 레벨 10/20/30/40 달성 시 AttackData 단계만 교체하면 데미지, 범위, 이펙트가 함께 바뀌도록 만듭니다.

AI몬스터 AI / 스폰 / 드랍

모든 몬스터는 같은 AI 구조를 공유하고, 구역별 데이터만 다르게 넣는 구조가 적합합니다.

  • MonsterData ScriptableObject: HP, 공격력, 경험치, 골드, 감지 범위, 드랍 테이블, 레벨을 정의합니다.
  • MonsterAI: Idle, Patrol, Chase, Attack, Dead 상태 머신으로 구성합니다.
  • MonsterSpawner: 구역별 스폰 포인트와 몬스터 종류를 연결합니다.
  • ItemDropper: 사망 이벤트를 받아 골드, 경험치, 퀘스트 아이템, 장비 드랍을 처리합니다.

Growth레벨 / 경험치 / 스탯

레벨 시스템은 다른 시스템의 해금 조건이므로 이벤트 기반으로 만들어두는 편이 좋습니다.

  • PlayerStats: 현재 HP, MP, EXP, Level, Gold, 공격 강화 단계를 보관합니다.
  • LevelTable: 레벨별 필요 경험치와 구역 권장 레벨을 데이터로 분리합니다.
  • OnLevelUp 이벤트: HUD 갱신, 공격 강화 해금, 퀘스트 조건 갱신을 연결합니다.
  • 퀘스트 완료 시 5레벨 상승, 몬스터 사냥으로 나머지 5레벨을 채우는 규칙을 데이터로 고정합니다.

Items인벤토리 / 장비 / 상점

일정상 가장 큰 리스크입니다. 먼저 20칸 보관, 자동 습득, 사용, 장착만 안정화하고 고급 연출은 뒤로 미루는 것이 좋습니다.

  • ItemData ScriptableObject: 아이콘, 타입, 가격, 스탯 보너스, 사용 효과, 장비 프리팹 정보를 가집니다.
  • InventoryManager: 20칸 배열, 골드, 아이템 추가/삭제/교환을 담당합니다.
  • InventorySlot UI: IBeginDragHandler, IDragHandler, IEndDragHandler, IDropHandler로 드래그 앤 드롭을 구현합니다.
  • EquipmentManager: 착용 슬롯과 캐릭터 모델 소켓을 연결해 장비 비주얼을 교체합니다.
  • ShopManager: 상점 타입별 판매 목록과 구매 가능 여부를 처리합니다.

Quest퀘스트 / NPC 상호작용

퀘스트는 선형 진행의 뼈대입니다. NPC와 HUD가 같은 QuestManager 상태를 바라보도록 구성해야 합니다.

  • QuestData ScriptableObject: 목표 타입, 목표 수량, 요구 아이템, 보상 골드, 다음 퀘스트를 정의합니다.
  • QuestManager: 현재 활성 퀘스트 하나만 보관하고 수락, 진행, 완료 상태를 관리합니다.
  • NPCQuestGiver: 느낌표/물음표 마커, F 상호작용, 대화 UI 열기를 담당합니다.
  • QuestProgressHUD: 우측 상단에 현재 목표와 진행도를 실시간으로 표시합니다.
  • Kill, Collect, Bring 타입을 공통 인터페이스로 처리하면 이후 확장이 쉽습니다.

World월드 / 구역 / 웨이포인트

노션 문서상 맵은 에셋 Demo/Prefab을 이어붙이는 방향으로 축소되어 있습니다. 단일 씬에 5개 구역을 배치하는 구조가 현실적입니다.

  • ZoneTrigger: 시작 마을, 황무지, 어둠 숲, 오크 주둔지, 왕의 무덤 진입을 감지합니다.
  • ZoneManager: BGM, 몬스터 스폰, 미니맵 표시, 권장 레벨 정보를 갱신합니다.
  • WaypointManager: 해금된 웨이포인트 목록과 순간이동 위치를 관리합니다.
  • 맵 배치 확정 후 NavMesh를 전체 베이크하고, Static/Occlusion 설정을 같이 정리합니다.

UIHUD / 게임오버 / 저장

UI는 여러 시스템의 현재 상태를 보여주는 층으로 두고, 직접 게임 로직을 갖지 않게 구성합니다.

  • HUDController: HP, MP, 포션 수량, EXP, 레벨, 퀘스트 목표를 표시합니다.
  • GameOverController: 사망 시 검은 화면, 카운트다운, 가장 가까운 웨이포인트 부활을 처리합니다.
  • SaveLoadManager: JSON으로 레벨, 경험치, 골드, 인벤토리, 장비, 퀘스트, 웨이포인트 해금 상태를 저장합니다.
  • UI는 데이터 변경 이벤트를 구독하고, 매 프레임 직접 값을 폴링하지 않도록 합니다.

Boss보스 / 최종 폴리싱

보스는 패턴 수보다 명확한 예고, 회피 가능성, 피드백이 중요합니다.

  • BossAI: 패턴 선택, 페이즈 전환, 사망 이벤트를 담당합니다.
  • PatternData: 광역 내려찍기, 직선 돌진, 소환 정도를 데이터화합니다.
  • 인디케이터, 카메라 흔들림, 피격음, 이펙트를 최소 세트로 묶어 타격감을 보강합니다.
  • D19-D20 일정에는 타이틀, SFX, 크리티컬 버그 수정, 빌드 검증을 우선 배치합니다.

추천 구현 순서

단계 목표 완료 기준
1 플레이어 이동, 대시, 카메라, 기본 공격 플레이어가 이동/대시/공격하고 애니메이션과 히트박스가 연결됩니다.
2 몬스터 AI와 데미지 루프 몬스터가 배회, 추적, 공격, 사망하며 경험치와 드랍을 줍니다.
3 레벨과 공격 강화 레벨업에 따라 공격 단계가 바뀌고 데미지/범위/이펙트가 달라집니다.
4 인벤토리와 아이템 습득 20칸 인벤토리, 자동 습득, 소모품 사용, 쓰레기통 삭제가 작동합니다.
5 퀘스트와 NPC NPC가 퀘스트를 주고, 목표 진행도가 HUD에 표시되며, 완료 보상이 지급됩니다.
6 월드 구역, 웨이포인트, 상점 5개 구역의 이동 흐름과 순간이동, 기본 구매 루프가 연결됩니다.
7 보스, 저장, 게임오버, 폴리싱 최종 보스 전투, 저장/로드, 부활, 타이틀, SFX, 빌드 테스트가 완료됩니다.

리스크와 축소 기준

리스크 대응 방식
인벤토리 + 장비 비주얼 교체 공수 증가 먼저 아이콘 기반 장착과 스탯 반영만 완성하고, 실제 모델 교체는 핵심 장비 몇 개로 제한합니다.
퀘스트와 NPC 연동 복잡도 동시에 여러 퀘스트를 받는 구조를 피하고, 활성 퀘스트 하나만 처리합니다.
5개 구역 맵 배치와 NavMesh 문제 초반에 스케일 기준을 고정하고, 구역 경계 Trigger와 스폰 포인트부터 먼저 배치합니다.
보스 패턴 과다 패턴은 2-3개로 제한하고, 예고/피격/사운드 피드백을 우선합니다.
