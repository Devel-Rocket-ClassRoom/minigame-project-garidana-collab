현재 구현 상태

완료

  • 웨이포인트 첫 상호작용 시 해금됩니다.
  • 해금되면 해당 웨이포인트의 포탈 비주얼이 활성화됩니다.
  • 마지막으로 활성화한 웨이포인트 ID를 WaypointManager가 기억합니다.
  • 플레이어 사망 시 마지막 활성 웨이포인트 또는 기본 부활 위치로 리스폰할 수 있습니다.
  • 1차 텔레포트 버전이 구현되어 있습니다.

1차 텔레포트 규칙

  • 시작 마을 웨이포인트(Town)와 상호작용하면 해금된 첫 번째 구역 웨이포인트(Region)로 즉시 이동합니다.
  • 구역 웨이포인트와 상호작용하면 해금된 첫 번째 시작 마을 웨이포인트로 즉시 복귀합니다.
  • 목적지 선택 UI는 아직 없습니다.

현재 한계

미구현

  • 여러 구역이 해금되어 있어도 시작 마을에서 목적지를 선택할 수 없습니다.
  • 세이브/로드에 해금 웨이포인트와 마지막 활성 웨이포인트가 아직 연결되지 않았습니다.
  • Unity 플레이모드 기준 전체 동작은 사용자가 직접 확인해야 합니다. Codex가 실제 플레이 검증까지 하지는 못했습니다.

작업 흐름 요약

1. 해금 방식 확정 퀘스트 해금이 아니라, 플레이어가 웨이포인트를 처음 상호작용하면 포탈이 켜지고 그 시점부터 사용 가능하도록 변경했습니다.
2. 상태 관리 분리 해금 여부와 마지막 활성 웨이포인트 기록은 WaypointManager가 담당하게 했습니다.
3. 공용 상호작용 흐름 유지 웨이포인트는 IInteractable 구현체로 처리해서 기존 F 상호작용 구조에 얹었습니다.
4. 부활 시스템 연동 사망 후 GameOverUi에서 리스폰 버튼을 누르면 마지막 활성 웨이포인트 또는 기본 부활 위치로 복귀합니다.
5. 1차 텔레포트 추가 시작 마을 허브와 구역 복귀만 가능한 최소 버전 텔레포트를 구현했습니다.

핵심 파일과 역할

Assets/Scripts/Waypoint/WaypointNode.cs
웨이포인트 한 개의 데이터 보관용입니다. WaypointId, DisplayName, WaypointType, SpawnPoint, PortalVisual를 가집니다.
Assets/Scripts/Waypoint/WaypointInteractable.cs
기존 상호작용 시스템에 붙는 진입점입니다. 잠겨 있으면 해금하고, 열려 있으면 목적지를 찾아 텔레포트합니다.
Assets/Scripts/Waypoint/WaypointManager.cs
웨이포인트 등록, 해금 상태, 마지막 활성 웨이포인트, 기본 부활 위치, 1차 텔레포트 목적지 선택을 담당합니다.
Assets/Scripts/Player/PlayerStats.cs
기존 사망/부활 로직이 들어 있는 플레이어 상태 스크립트입니다. 여기에 RespawnAt()과 별도로 TeleportTo()를 추가해서 웨이포인트 이동을 분리했습니다.
Assets/Scripts/UI/GameOverUi.cs
게임오버 패널과 리스폰 버튼 제어를 담당합니다. 자동 리스폰은 제거했고, 일정 시간 뒤 버튼을 눌러 수동 리스폰하는 구조입니다.
Assets/Scripts/PlayerInteractor.cs
죽었다가 리스폰한 뒤 상호작용이 완전히 막히던 버그를 이 파일에서 수정했습니다. 이제 컴포넌트를 끄지 않고 IsDead 기준으로만 상호작용을 막습니다.

Unity 인스펙터 세팅 체크리스트

  • 각 웨이포인트 오브젝트에 WaypointNode와 WaypointInteractable 둘 다 붙어 있어야 합니다.
  • WaypointNode._waypointId는 반드시 웨이포인트마다 고유해야 합니다.
  • 시작 마을은 Waypoint Type = Town, 구역 웨이포인트는 Waypoint Type = Region으로 설정해야 합니다.
  • Portal Visual에는 실제 포탈 이펙트 또는 메시 오브젝트를 연결해야 합니다.
  • Spawn Point가 비어 있으면 웨이포인트 자기 자신의 위치를 사용합니다.
  • 웨이포인트에 상호작용 판정용 Collider가 있어야 하고, PlayerInteractor가 탐지하는 레이어에 포함되어야 합니다.
  • 씬의 WaypointManager 오브젝트에는 모든 웨이포인트가 _waypoints 리스트에 등록되어 있어야 합니다.
  • 웨이포인트를 하나도 안 찍고 죽었을 때를 대비해 WaypointManager._defaultRespawnPoint를 연결해 두는 것이 좋습니다.

지금까지 확인된 이슈와 원인

  • 두 번째 웨이포인트가 잠긴 상태인데도 Use Waypoint로 보이던 문제는 잘못된 waypointId 입력이 원인이었습니다.
  • 죽고 리스폰한 뒤 포탈, 힐러, NPC 상호작용 UI가 아예 안 뜨던 문제는 PlayerInteractor가 죽을 때 enabled = false로 꺼지던 구조 때문이었습니다.
  • 현재는 PlayerInteractor를 끄지 않고, 죽은 상태에서만 내부 로직을 멈추는 방식으로 수정했습니다.
다른 PC에서 이어받는 경우, 먼저 Unity에서 Waypoint Type, WaypointId, WaypointManager._waypoints, Default Respawn Point 네 가지가 제대로 연결되어 있는지 확인하는 것이 가장 빠릅니다.

현재 동작 흐름

1. 플레이어가 웨이포인트 근처에서 F 입력
2. PlayerInteractor가 IInteractable 대상으로 WaypointInteractable을 호출
3. 아직 잠겨 있으면:
   - WaypointManager.UnlockWaypoint()
   - 포탈 비주얼 활성화
   - 마지막 활성 웨이포인트 갱신
4. 이미 해금된 상태면:
   - 현재 웨이포인트 타입 확인
   - Town이면 해금된 첫 Region 찾기
   - Region이면 해금된 첫 Town 찾기
   - PlayerStats.TeleportTo()로 즉시 이동
   - 목적지 웨이포인트를 마지막 활성 웨이포인트로 갱신
5. 플레이어가 죽으면:
   - GameOverUi 표시
   - 일정 시간 후 리스폰 버튼 활성화
   - 버튼 클릭 시 WaypointManager.GetRespawnPoint() 위치로 RespawnAt()

전달 메모

이 작업은 기존 프로젝트의 공용 F 상호작용 구조를 유지하는 방향으로 진행했습니다. 따라서 웨이포인트 전용 입력 시스템을 새로 만들지 않았고, PlayerInteractor → IInteractable → WaypointInteractable → WaypointManager 구조를 유지하는 것이 중요합니다.
