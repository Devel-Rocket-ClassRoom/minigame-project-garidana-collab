Inventory Implementation Record

인벤토리 시스템 구현 정리

이 문서는 현재 프로젝트에서 인벤토리 기능이 어떤 파일에, 어떤 책임으로, 어떤 흐름으로 구현되어 있는지 정리한 기록입니다. 목적은 단순 기능 목록이 아니라, 나중에 장착 해제, 상점, 정렬, 비교 툴팁, 저장/불러오기 같은 기능을 추가할 때 기존 구조를 빠르게 이해하고 안전하게 확장할 수 있게 만드는 데 있습니다.

대상 범위
입력, 보유 아이템, 장착 상태, 인벤토리 UI, 슬롯 상호작용, 툴팁 UI
기준 시점
2026-05-31 현재 작업 트리 기준
핵심 원칙
데이터 원본과 화면 표현을 분리하고, UI는 가능한 한 상태를 반영하는 역할만 담당
전체 개요 입력 처리 데이터 구조 장착 시스템 인벤토리 UI 슬롯 상호작용 툴팁 시스템 씬 연결 포인트 확장 포인트 수정 체크리스트

1. 전체 개요

현재 인벤토리 시스템은 한 파일에서 모든 것을 처리하는 구조가 아니라, 입력 → 데이터 상태 → 장착 상태 → UI 갱신 → 슬롯 상호작용 → 툴팁 표시로 층이 나뉘어 있습니다. 이 구조 덕분에 UI를 바꾸더라도 장착 로직 전체를 건드릴 필요가 없고, 반대로 장착 규칙을 바꾸더라도 툴팁 시스템을 다시 짤 필요가 없습니다.

STEP 01 입력
PlayerInputReader가 Input System의 Inventory 액션을 받아 InventoryPressed 이벤트를 발생시킵니다.
↓
STEP 02 패널 제어
InventoryUi가 이벤트를 구독하고 패널을 열거나 닫습니다. 열릴 때는 슬롯, 스탯, 현재 장착 UI를 함께 갱신합니다.
↓
STEP 03 데이터 조회
PlayerInventory에서 보유 아이템을, EquipmentManager에서 현재 장착 상태를, PlayerStats와 QuestManager에서 부가 정보를 읽습니다.
↓
STEP 04 슬롯 생성과 상호작용
InventoryUi가 부위별 행에 슬롯 프리팹을 생성하고, 각 슬롯은 InventorySlotUi를 통해 hover, 우클릭 장착, 선택 하이라이트를 처리합니다.
↓
STEP 05 공용 툴팁 표시
ItemTooltipUi가 같은 Canvas 아래에서 툴팁과 액션 힌트를 공용으로 관리합니다. 이 구조는 상점 UI에도 재사용할 수 있습니다.
보유 데이터의 원천
PlayerInventory의 아이템 목록입니다.
장착 상태의 원천
EquipmentManager의 현재 장착 필드와 장착 이벤트입니다.
표시 규칙의 원천
InventoryUi가 행 구성, 슬롯 수, 아이콘 갱신 규칙을 담당합니다.

2. 입력 처리

인벤토리 열기/닫기는 더 이상 Update()에서 직접 키를 읽는 방식이 아닙니다. 현재 구조는 Input System 등록 액션을 PlayerInputReader가 받고, UI는 그 이벤트만 구독하는 구조입니다.

구성 요소 현재 구현 방식 의도
PlayerInputReader InventoryPressed 이벤트와 OnInventory(InputValue value)를 통해 입력을 수신합니다. 입력 시스템과 개별 UI 스크립트를 분리하기 위함입니다.
InventoryUi PlayerInputReader.InventoryPressed += TogglePanel 형태로 구독합니다. UI가 직접 키를 모르고도 열고 닫히게 하기 위함입니다.
패널 토글 TogglePanel()과 내부 SetPanelActive()에서 처리합니다. 열고 닫을 때 함께 갱신하거나 숨겨야 하는 로직을 한 곳에 모으기 위함입니다.
이 구조의 장점은, 나중에 인벤토리 외에 정렬 단축키, 장착 비교 모드, 상점 열기 같은 기능을 추가할 때도 입력 경로를 PlayerInputReader 중심으로 통일할 수 있다는 점입니다.

3. 데이터 구조

인벤토리에 표시되는 대부분의 값은 실제로 여러 소스에서 읽어옵니다. 따라서 기능을 수정할 때는 “어디가 진짜 상태 원본인지”를 먼저 아셔야 합니다.

3-1. 보유 아이템 데이터

보유 아이템 자체는 Assets/Scripts/Player/PlayerInventory.cs가 담당합니다. 인벤토리 보유 목록 UI는 여기 있는 아이템을 기본으로 읽습니다.

경로: Assets/Scripts/Player/PlayerInventory.cs

3-2. 장착 상태 데이터

현재 어떤 장비를 장착 중인지의 원본은 Assets/Scripts/Player/EquipmentManager.cs 입니다. 지금은 다음 5부위를 관리합니다.

  • Sword
  • Shield
  • Helmet
  • Chest
  • Legs

경로: Assets/Scripts/Player/EquipmentManager.cs

3-3. 아이템 정의 데이터

아이템 하나하나의 표시 이름, 설명, 티어, 보너스 수치, 아이콘, 장착 대상 모델 이름은 모두 ItemData ScriptableObject에서 읽습니다. 현재 확인되는 핵심 필드는 다음과 같습니다.

  • displayName
  • description
  • tier
  • attackBonus
  • maxHealthBonus
  • icon
  • equipObjectName

경로: Assets/Scripts/ScriptableObjects/ItemData.cs

3-4. 스탯과 퀘스트 데이터

좌측 스탯 영역은 인벤토리만의 별도 데이터가 아니라, 기존 플레이어 상태를 그대로 읽어오는 방식입니다.

  • PlayerStats에서 레벨, 공격력, 현재/최대 체력을 읽습니다.
  • PlayerAttackUpgrade에서 현재 강화 단계를 읽습니다.
  • QuestManager.CompletedQuests에서 완료 퀘스트 목록을 읽습니다.

4. 장착 시스템

장착 시스템의 핵심 역할은 “현재 장착 데이터 보관”과 “실제 플레이어 모델 on/off 전환”입니다. 이 둘을 EquipmentManager가 같이 책임지고 있습니다.

4-1. 기본 장비 적용

시작 시 기본 장비는 EquipDefaultItem() 흐름으로 처리됩니다. 현재 구현에서는 기본 장비를 단순히 장착만 하는 것이 아니라, 필요 시 PlayerInventory에도 넣은 뒤 장착하도록 보강되어 있습니다. 이 덕분에 “현재 장착 중인 장비도 보유 장비 목록에 보여야 한다”는 요구를 충족할 수 있습니다.

4-2. 현재 장착 조회

InventoryUi는 EquipmentManager.GetEquippedItem(EquipmentPart part)를 통해 각 부위의 현재 장비를 조회합니다. 즉 UI는 장착 상태를 따로 저장하지 않고, 장착 관리자에게 물어보는 구조입니다.

4-3. 모델 활성화

장비를 장착하면 equipObjectName을 기준으로 플레이어 하위 모델을 찾아 켜고, 이전 장비 모델은 끄는 방식으로 보이는 외형이 바뀝니다. 현재 코드는 mount parent를 먼저 보고, 없으면 플레이어 하위 전체에서 한 번 더 찾도록 보완되어 있습니다.

4-4. 장착 변경 이벤트

장착 변경 후 EquipmentChanged 이벤트를 발생시키고, InventoryUi는 이를 구독해서 슬롯 하이라이트와 현재 장착 아이콘을 다시 갱신합니다. 따라서 UI 갱신의 핵심 트리거는 “장착 변경 이벤트”입니다.

현재 구조에서는 장착 상태가 여러 곳에 중복 저장되지 않도록 유지하는 것이 중요합니다. 새 기능을 넣을 때 InventoryUi 안에 별도의 장착 캐시를 만들면 쉽게 어긋날 수 있으므로, 가능하면 항상 EquipmentManager를 진실한 원천으로 유지하는 편이 안전합니다.

5. 인벤토리 UI 구현

실제 화면 조립과 갱신의 중심은 Assets/Scripts/UI/InventoryUi.cs 입니다. 이 스크립트는 단순 패널 on/off만 하는 것이 아니라, 인벤토리 화면 전체의 오케스트레이터 역할을 합니다.

경로: Assets/Scripts/UI/InventoryUi.cs

5-1. 패널 열기와 닫기

TogglePanel()이 패널 활성 상태를 바꾸고, 열릴 때는 즉시 다음 정보를 다시 읽습니다.

  • RefreshSlots()
  • RefreshStatSection()

닫을 때는 툴팁이 남지 않도록 ItemTooltipUi.HideTooltip()도 함께 호출합니다.

5-2. 스탯 영역 갱신

좌측 Stat 영역은 RefreshStatSection()에서 갱신합니다. 현재 표시 대상은 다음과 같습니다.

  • 레벨
  • 현재 공격 강화 단계
  • 최종 공격력
  • 현재 체력 / 최대 체력
  • 완료 퀘스트 목록

완료 퀘스트는 QuestManager.Instance.CompletedQuests를 읽고, 비어 있는 칸은 -로 표시합니다.

5-3. 현재 장착 영역 갱신

가운데 CurrentEquip 영역은 배경 이미지와 실제 장비 아이콘을 분리하는 방식으로 구현되어 있습니다. 각 슬롯 배경 위에 런타임에서 EquippedIcon 자식 Image를 만들어, 배경은 유지하고 아이콘만 바꾸도록 되어 있습니다.

이 방식의 장점은, 장비 칸의 프레임 디자인을 유지하면서 현재 장착 아이콘만 갈아끼울 수 있다는 점입니다.

5-4. 보유 장비 목록 구성

오른쪽 보유 장비 영역은 한 리스트가 아니라 5개 행으로 분리되어 있습니다.

행 부위 의미
1행 Sword 검 장비만 표시
2행 Shield 방패 장비만 표시
3행 Helmet 머리 장비만 표시
4행 Chest 가슴 장비만 표시
5행 Legs 다리 장비만 표시

각 행은 현재 _slotsPerRow = 6을 기본으로 사용합니다. 즉 아이템이 1개만 있어도 6칸 전체가 유지되고, 첫 슬롯에 아이템이 들어가며 나머지는 빈 슬롯으로 남습니다. 이 점이 초기 “아이템 하나가 행 전체를 가득 채우던 문제”를 해결한 부분입니다.

5-5. 현재 장착 아이템도 보유 목록에 포함

CollectOwnedEquipmentItems()는 단순히 PlayerInventory만 읽는 것이 아니라, 현재 장착 중인 장비가 목록에 없으면 EquipmentManager에서 읽어와 행 목록에 보강합니다. 그래서 기본 장비처럼 “장착은 되어 있지만 인벤토리 표시에서는 빠질 수 있었던 경우”도 보완됩니다.

현재 UI 구조는 “행별 부위 구분”과 “항상 고정 칸 수 유지”가 핵심입니다. 따라서 나중에 정렬 기능을 넣더라도 각 행 안에서만 정렬하면 되고, 전체 장비를 한 리스트에서 다시 분류할 필요가 없습니다.

6. 슬롯 상호작용

슬롯 한 칸 자체의 동작은 Assets/Scripts/UI/InventorySlotUi.cs가 담당합니다. 이 스크립트는 “아이콘 보여주기”만 하는 것이 아니라, hover, 클릭, 선택 상태까지 담당하는 단위 컴포넌트입니다.

경로: Assets/Scripts/UI/InventorySlotUi.cs

6-1. 슬롯 구성 요소

현재 슬롯 프리팹은 대체로 다음 역할로 나뉩니다.

  • BackgroundImage: 부위별 슬롯 배경
  • IconImage: 실제 아이템 아이콘
  • Button: 클릭 이벤트 수신
  • EquippedHighlight: 현재 장착 슬롯 표시
  • NameText: 현재는 필수 아님, 툴팁 방식이면 비워둘 수 있음

6-2. 슬롯 세팅 방식

Setup(ItemData data, bool isEquipped, Action<ItemData> onClicked, Sprite backgroundSprite = null) 형태로 데이터를 주입합니다. 이때 슬롯은 다음을 동시에 결정합니다.

  • 어떤 아이템을 나타내는지
  • 장착 중인지
  • 어떤 배경 이미지를 쓸지
  • 클릭이 가능한 슬롯인지

6-3. 포인터 이벤트

슬롯은 IPointerEnterHandler, IPointerExitHandler, IPointerMoveHandler, IPointerClickHandler를 구현합니다. 그래서 hover와 우클릭 장착이 슬롯 단위에서 처리됩니다.

이벤트 현재 동작
OnPointerEnter 툴팁 표시, 액션 힌트 표시
OnPointerMove 마우스 위치에 맞춰 툴팁 위치 갱신
OnPointerExit 툴팁 숨김
OnPointerClick 우클릭이면 장착 처리

현재 액션 힌트 문구는 코드상 [Right Click to Equip]로 전달되고 있습니다. 즉 기능은 정상이나, 나중에 한글 문구로 정리하거나 장착/판매/구매 같은 상황별 문구로 분리하는 것은 좋은 후속 작업이 될 수 있습니다.

7. 툴팁 시스템

툴팁은 인벤토리 전용 오브젝트를 씬에 미리 만들어두는 방식이 아니라, Assets/Scripts/UI/ItemTooltipUi.cs가 같은 Canvas 아래에서 런타임에 공용 UI를 관리하는 방식입니다.

경로: Assets/Scripts/UI/ItemTooltipUi.cs

7-1. 왜 별도 공용 시스템으로 분리했는가

인벤토리 슬롯 내부에 설명창을 직접 박아두면, 상점이나 드랍 아이템 UI에서 같은 정보를 다시 구현해야 합니다. 현재처럼 공용 툴팁 시스템으로 분리하면, 나중에 다른 UI도 ItemTooltipUi.Show(item, position, hint)만 호출하면 같은 툴팁을 재사용할 수 있습니다.

7-2. 표시 정보

현재 툴팁은 다음 정보를 ItemData에서 읽습니다.

  • Display Name
  • Description
  • Tier
  • Attack Bonus
  • Max Health Bonus

7-3. 위치 계산

현재 툴팁은 기본적으로 마우스 포인터 기준 우상단에 뜨도록 계산됩니다. 다만 화면 위쪽이나 오른쪽 경계를 넘어가면 아래쪽 또는 왼쪽으로 반전합니다.

  • 기본: 우상단
  • 위쪽 넘침: 우하단
  • 오른쪽 넘침: 좌상단
  • 둘 다 넘침: 좌하단

7-4. Canvas 귀속 문제 해결

초기에 툴팁이 생성은 되지만 화면에 안 보이던 원인은, 잘못된 Canvas 아래 생성되거나 다른 UI 뒤에 깔리는 문제였습니다. 현재는 EnsureForCanvas(Canvas preferredCanvas)를 통해 인벤토리 UI와 같은 Canvas에 툴팁을 귀속시키고, sibling 순서도 마지막으로 올려 다른 UI 위에 보이게 처리하고 있습니다.

7-5. 인벤토리 닫힘과 툴팁 숨김

현재는 인벤토리를 닫을 때 InventoryUi가 ItemTooltipUi.HideTooltip()를 같이 호출합니다. 그래서 hover 상태에서 Tab으로 패널을 닫아도 툴팁과 우하단 액션 힌트가 남지 않도록 처리되어 있습니다.

우하단에 따로 뜨는 메시지는 툴팁 본문이 아니라 별도 ActionHint 패널입니다. 이 구조는 상점에서 구매, 인벤토리에서는 장착, 판매 UI에서는 판매처럼 상황별 액션 안내를 재사용하기 쉽게 만들기 위한 선택입니다.

8. 씬 연결 포인트

현재 구조는 코드만으로 완성되지 않고, 씬과 프리팹 연결이 매우 중요합니다. 특히 InventoryUi와 슬롯 프리팹은 인스펙터 연결이 틀리면 코드가 맞아도 보이지 않습니다.

InventoryUi에서 연결해야 하는 주요 필드
  • _panelRoot: 실제 인벤토리 패널
  • _swordSlotRow ~ _legsSlotRow: 5개 행 컨테이너
  • _slotPrefab: 공통 슬롯 프리팹 1개
  • _swordSlotBackground ~ _legsSlotBackground: 부위별 배경 스프라이트
  • _playerStats, _playerAttackUpgrade, 장착 아이콘, TMP 텍스트들
슬롯 프리팹에서 연결해야 하는 주요 필드
  • _backgroundImage: 슬롯 틀 이미지
  • _iconImage: 실제 아이템 아이콘 이미지
  • _button: 루트 버튼
  • _equippedHighlight: 장착 하이라이트
  • _nameText: 선택 사항
인벤토리 토글 컨트롤러와 실제 패널은 같은 오브젝트가 아니어야 합니다. 컨트롤러까지 비활성화되면 입력 이벤트 구독이 끊겨 다시 켤 수 없기 때문입니다.

9. 확장 포인트

현재 구조는 이미 다음 기능들을 붙이기 좋은 상태로 나뉘어 있습니다. 다만 어디를 확장해야 하는지만 명확히 아시면 불필요한 중복 로직을 만들지 않을 수 있습니다.

9-1. 장착 해제 기능

추천 위치는 EquipmentManager입니다. 장착 해제는 단순 UI 제거가 아니라 장착 상태와 플레이어 모델 on/off까지 함께 바꿔야 하므로, InventoryUi가 직접 처리하는 구조보다 EquipmentManager에 전용 메서드를 추가하는 편이 맞습니다.

9-2. 상점 연동

현재 툴팁 시스템은 이미 공용화되어 있으므로 상점에서 그대로 재사용할 수 있습니다. 상점에서는 같은 ItemTooltipUi.Show()를 쓰되, 액션 힌트를 구매나 판매로 바꾸는 식으로 확장하는 것이 자연스럽습니다.

9-3. 비교 툴팁

hover 중인 아이템과 현재 장착 중인 같은 부위 아이템의 attackBonus, maxHealthBonus를 비교해 증감치를 보여주려면, ItemTooltipUi에 현재 장착 아이템 컨텍스트를 추가로 넘기는 방식이 가장 깔끔합니다.

9-4. 정렬과 필터

현재는 이미 부위별 행 분리가 되어 있으므로, 전체 아이템을 다시 분해할 필요 없이 “행 내부 정렬”만 넣어도 됩니다. 예를 들어 티어 높은 순, 공격력 높은 순 같은 정렬은 CollectOwnedEquipmentItems() 이후 행별 리스트에 적용하는 쪽이 자연스럽습니다.

9-5. 소비 아이템 / 퀘스트 아이템 탭 분리

장비 외 아이템이 늘어날 경우 현재 장비 행 구조에 그대로 섞기보다, 카테고리 탭을 나누는 쪽이 더 확장성이 좋습니다. 이때 InventoryUi는 “현재 선택된 카테고리” 개념만 추가하고, 슬롯 프리팹은 그대로 재사용할 수 있습니다.

9-6. 저장 / 불러오기

세이브 시스템을 붙일 때는 PlayerInventory와 EquipmentManager를 함께 저장해야 합니다. 하나만 저장하면 “장착은 되어 있는데 인벤토리에 없음”, 혹은 반대로 “인벤토리엔 있는데 장착 상태는 초기화됨” 같은 어긋남이 생길 수 있습니다.

10. 나중에 수정할 때 보는 체크리스트

증상 가장 먼저 볼 파일 확인 포인트
Tab을 눌러도 인벤토리가 안 열림 PlayerInputReader.cs, InventoryUi.cs Inventory 액션 연결, 이벤트 구독, 컨트롤러 오브젝트 비활성화 여부
보유 장비 슬롯이 안 뜸 InventoryUi.cs 행 컨테이너 연결, 슬롯 프리팹 연결, 행별 생성 로직
현재 장착 아이콘이 이상함 InventoryUi.cs, EquipmentManager.cs 현재 장착 조회, 아이콘 생성, 기본 장비 설정
머리/가슴/다리 장착이 안 바뀜 EquipmentManager.cs, 해당 ItemData asset equipObjectName, 기본 장비 연결, 실제 플레이어 하이어라키 이름
툴팁이 안 뜨거나 화면 밖으로 나감 ItemTooltipUi.cs, InventorySlotUi.cs Canvas 귀속, sibling 순서, 포인터 이벤트, 위치 반전 계산
툴팁이 닫힌 뒤에도 남음 InventoryUi.cs 패널 닫힘 시 HideTooltip() 호출 여부
현재 인벤토리 시스템의 가장 중요한 구조적 장점은, “데이터 원본”, “장착 상태”, “화면 표현”, “슬롯 상호작용”, “툴팁”이 한 덩어리로 섞여 있지 않다는 점입니다. 앞으로 기능을 추가하실 때도 이 분리를 유지하시면 수정 범위를 좁게 가져가기가 훨씬 쉬워집니다.
현재 문서는 실제 구현 기준으로 작성되었기 때문에, 일부 문구나 디자인 요소는 임시 상태가 남아 있을 수 있습니다. 예를 들어 슬롯 액션 힌트 문구는 현재 코드상 [Right Click to Equip]입니다. 기능 확장 시에는 이런 사용자 문구도 함께 정리하시는 편이 좋습니다.
